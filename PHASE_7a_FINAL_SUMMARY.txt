KEY TECHNICAL DECISIONS

1. scred-config as Separate Crate
   Rationale: Centralized configuration management for all three binaries
   Benefits: No circular dependencies, clean architecture, code reuse

2. 6-Level File Precedence
   Precedence: /etc/scred → ~/.scred → scred-{ENV} → ./scred → $SCRED_CONFIG_FILE → defaults
   Rationale: Balances flexibility (user overrides) with security (admin defaults)

3. Per-Path Rules Only in scred-proxy
   Rationale: Full selective redaction is unique to reverse proxy use case
   Benefit: MITM always redacts based on mode (simpler)

4. Hot-Reload with SIGHUP (Unix) + No-Op (Windows)
   Rationale: Follows Unix conventions, graceful degradation on Windows
   Benefit: Operators already know about SIGHUP signal

5. Async Tokio Implementation
   Rationale: All binaries use tokio already, native signal support
   Benefit: Efficient, no blocking, integrates cleanly
