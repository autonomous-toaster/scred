CURRENT ARCHITECTURE PROBLEMS
═══════════════════════════════════════════════════════════════════════════════

Problem 1: Three Redaction Engines with Different Capabilities
  ❌ RedactionEngine: No selector support
  ❌ StreamingRedactor: No selector support
  ❌ ConfigurableEngine: Has selector support, but CLI-only
  
Result: Inconsistent behavior

Problem 2: Code Duplication Prevents Consistent Fixes
  ❌ HTTP proxy logic in 3 places (proxy main, http_proxy_handler, MITM wrapper)
  ❌ Each location has different engine
  ❌ Updates to one don't apply to others
  
Result: Hard to fix consistently

Problem 3: Dead Code and Unused Variables
  ❌ Proxy creates _config_engine (unused)
  ❌ MITM H2 passes selector parameters (never used)
  ❌ Comments document broken selector support
  
Result: Technical debt and confusion

Problem 4: Type System Constraints Block Refactoring
  ❌ http_proxy_handler expects Arc<RedactionEngine>
  ❌ Can't pass Arc<ConfigurableEngine> without trait objects
  ❌ Would require risky refactoring of shared code
  
Result: Blocked from using existing solution

═══════════════════════════════════════════════════════════════════════════════

PROPOSED SOLUTION: EXTEND BASE ENGINES
═══════════════════════════════════════════════════════════════════════════════

Instead of:
  ❌ Creating wrapper abstractions
  ❌ Refactoring with trait objects
  ❌ Using ConfigurableEngine everywhere
  ❌ Duplicating code

Do this:
  ✅ Add selector support to RedactionEngine directly
  ✅ Add selector support to StreamingRedactor directly
  ✅ Make selector optional for backward compatibility
  ✅ Use everywhere without refactoring

CORE PRINCIPLE:
  Make engines selector-aware from the start, not as a wrapper.
  
This achieves:
  ✅ Single source of truth (selector logic in engines)
  ✅ No wrapper complexity
  ✅ Consistent everywhere
  ✅ Type-safe (no trait objects needed)
  ✅ First-class streaming support
  ✅ Clean architecture
