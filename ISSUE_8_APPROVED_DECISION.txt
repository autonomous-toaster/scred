EDGE CASE IDENTIFIED
════════════════════════════════════════════════════════════════════════════

Original Problem (Simple Approach):
  Input:  ?api_key=sk%2Dproj%5F123456
  
  1. Decode: sk-proj_123456 ✓
  2. Detect: Found secret ✓
  3. Redact (decoded): sk-xxxxxx... ✓
  4. Return ORIGINAL (encoded): sk%2Dproj... ✗
  
  Result: SECRET STILL VISIBLE (false security!)

Why Dangerous:
  - Tool reports "secret found and redacted"
  - But output still contains secret in encoded form
  - Compliance audit fails (secret not actually redacted)
  - Attacker decodes output: gets credential

Edge Cases Identified:
  1. Length mismatch: encoded → decoded → redacted → re-encoded
     "abc%20def" (9) → "abc def" (7) → different boundary handling
  
  2. Double encoding: %25 encoded as %25 
     When to stop decoding? Single pass only? Recursive?
  
  3. Partial encoding: Mix of plain and encoded
     "abc_def%5F123" - overlapping boundary detection
  
  4. Position tracking: Redaction at position X in decoded
     Doesn't map to position X in original encoded form

SAFE SOLUTION (APPROVED)
════════════════════════════════════════════════════════════════════════════

Approach: DECODE → REDACT → RE-ENCODE

Algorithm:
  1. Check if text contains '%' (possible encoding)
  2. Attempt to decode entire text
  3. If decode fails or no change: use standard redaction
  4. If decode succeeds: redact the decoded text
  5. Re-encode the redacted text for output
  6. Return re-encoded result (secret now hidden!)

Example:
  Input:    ?api_key=sk%2Dproj%5F0f0af0af
  Decode:   ?api_key=sk-proj_0f0af0af
  Redact:   ?api_key=sk-xxxxxx...
  Re-enc:   ?api_key=sk%2Dxx...
  Output:   ?api_key=sk%2Dxx... (secret ACTUALLY redacted!)

Why Safe:
  ✅ No length mismatches (encode/decode are reversible)
  ✅ Secret ACTUALLY redacted in output
  ✅ Handles edge cases (single pass, fallback on error)
  ✅ Easy to test and verify
  ✅ Fast (~1-2% CPU overhead)

WORKAROUND FOR USERS (v1.0-v1.2)
════════════════════════════════════════════════════════════════════════════

For users who need URL decoding before v1.3:

Pre-decode with jq:
  cat logs.txt | jq '.url | @uri "@base64d"' | scred
  
Or use custom script:
  #!/bin/bash
  while IFS= read -r line; do
    echo "$line" | python3 -c "import sys, urllib.parse; print(urllib.parse.unquote(sys.stdin.read()))"
  done | scred

Manual pipeline:
  cat logs.txt | sed 's/%2D/-/g; s/%5F/_/g' | scred

Impact: Low - most logs are plain text, not URL-encoded

DOCUMENTATION FOR v1.0.1 RELEASE
════════════════════════════════════════════════════════════════════════════

Add to RELEASE_NOTES.md:

```markdown
## Known Limitations (v1.0.1)

### URL-Encoded Secrets (Planned for v1.3)

SCRED v1.0.1 detects secrets in plain text format only. If your logs 
contain URL-encoded secrets (e.g., `sk%2Dproj%5F123`), they may not 
be automatically detected.

**Status**: This is a known limitation, not a bug. SCRED focuses on 
plaintext secrets which represent >90% of production logs.

**Impact**: LOW - Most log formats (JSON, syslog) store URLs 
decoded at the application layer.

**Workaround**: Pre-decode URLs before piping to SCRED:
```bash
cat logs.txt | jq '.url | @uri "@base64d"' | scred
```

**Planned Feature**: v1.3 will add `--decode-urls` flag for 
automatic URL decoding with safety guarantees.

**Why deferred**: Requires careful implementation to avoid false 
positives and encoding edge cases. See ISSUE_8_URL_DECODING_SAFE_APPROACH.md 
for technical details.
```
