🔴 #1: PROXY PER-PATH RULES NOT ENFORCED
   Location: crates/scred-proxy/src/main.rs
   Severity: CRITICAL
   Status: VERIFIED ✅
   
   Issue: Per-path redaction rules defined in config, parsed, stored
          BUT never checked during actual request processing
   
   Attack: curl http://localhost:9999/admin/secret?key=sk-123456
           Expected: sk-123456 NOT redacted (rule says "no redaction")
           Actual: sk-123456 redacted (rule ignored)
   
   Impact: Configuration false sense of security
   
   Fix: Add path-matching logic in request handler:
        ```
        if should_redact_for_path(&request_path, &per_path_rules) {
            redact_content()
        }
        ```

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🔴 #2: REGEX SELECTOR COMPLETELY BROKEN
   Location: crates/scred-http/src/pattern_selector.rs:192-200
   Severity: CRITICAL
   Status: VERIFIED ✅
   
   Issue: Code claims to implement "regex" selector but uses contains()
          
   Attack: --redact "regex:^sk-"
           Expected: Match pattern names starting with "sk-"
           Actual: Looks for pattern name containing literal "^sk-"
           Result: NO matches! Nothing redacted!
   
           Or: --redact "regex:sk-"
           Expected: Pattern names containing "sk-"
           Actual: Same behavior as intended
           But: Doesn't work as regex (no anchors, groups, etc)
   
   Impact: Feature marketed as available but completely non-functional
   
   Fix: Either implement actual regex with regex crate
        OR remove regex option and document as not supported

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🔴 #3: SILENT FALLBACK ON INVALID PATTERN SELECTOR
   Location: crates/scred-cli/src/main.rs:50-55
   Severity: CRITICAL
   Status: VERIFIED ✅
   
   Issue: Invalid selector triggers warning + silent fallback to default
          User never sees error; configuration silently changes behavior
   
   Attack: scred --redact CRIITICAL < file.txt  (typo in tier name)
           Expected: Error exit, config not applied
           Actual: warn!() logged to stderr
                   Falls back to default_redact()
                   User gets CRITICAL,API_KEYS,PATTERNS instead of just CRITICAL
                   If stderr redirected: warning lost
                   User unaware of policy deviation!
   
   Impact: User expectations violated silently; secrets exposed unexpectedly
   
   Fix: Change to error instead of fallback:
        ```rust
        let redact_selector = PatternSelector::from_str(redact_str)
            .map_err(|e| {
                eprintln!("ERROR: Invalid selector '{}': {}", redact_str, e);
                std::process::exit(1);
            })?;
        ```

🟠 #4: INCONSISTENT PATTERN SELECTOR PRECEDENCE
   
   CLI: --redact > SCRED_REDACT_PATTERNS > "CRITICAL,API_KEYS,PATTERNS"
   MITM: config_file > --redact > SCRED_REDACT_PATTERNS > Config::load()
   Proxy: config_file > SCRED_REDACT_PATTERNS (no --redact flag!)
   
   Result: Same deployment script behaves differently on each component
   Users confused when different redaction policies apply

🟠 #5: ENV-MODE DOESN'T VALIDATE PATTERN SELECTOR
   
   redact_env_line_configurable() passes engine but never validates selector
   User can't verify --redact CRITICAL actually applied to env mode
   Silent behavior changes if selector not properly passed

🟠 #6: TYPE CONFUSION IN PATTERN SELECTOR PARSING
   
   Input "aws-akia" is ambiguous:
   - Is it a tier name? (No)
   - Is it a pattern name? (Yes, in metadata)
   - Treated as wildcard pattern!
   
   Result: Inconsistent behavior if pattern name contains "-"
