 STARTING POINT
═══════════════════════════════════════════════════════════════════════════════

Your Constraint:
  "I'm OK with everything, EXCEPT having h2 specific in redactor"

Current Problem:
  ❌ h2_adapter in scred-http (protocol-specific in generic HTTP crate)
  ❌ scred-http mixing: protocols + h2_adapter + proxy handlers
  ❌ No clear separation between detection, redaction, and protocol handling
  ❌ Hard to reuse redaction logic outside MITM/proxy context
  ❌ Tight coupling between concerns


═══════════════════════════════════════════════════════════════════════════════
 PROPOSED SOLUTION
═══════════════════════════════════════════════════════════════════════════════

Create 2 new complementary HTTP-specific crates + refactor existing ones:

1. scred-http-detector (NEW)
   - Analysis layer (no mutations)
   - Detects what needs redaction
   - Classifies sensitivity levels
   - Protocol-agnostic

2. scred-http-redactor (NEW)
   - Redaction layer (actual mutations)
   - HTTP-specific redaction strategies
   - Includes H2Redactor (moved from h2_adapter)
   - Includes Http11Redactor
   - Streaming support for large bodies

3. scred-http (REFACTORED)
   - Remove h2_adapter (moved to redactor)
   - Keep protocol handlers
   - Simpler, focused scope

Result: 5-layer clean architecture with no cycles

═══════════════════════════════════════════════════════════════════════════════
 DATA FLOW (Request Processing Example)
═══════════════════════════════════════════════════════════════════════════════

Client sends: POST /api/login {"username": "john", "password": "secret123"}

  1. PARSE (scred-http::protocol::http1)
     ├─ Parse request line: POST /api/login HTTP/1.1
     ├─ Parse headers: Content-Type, Authorization, etc.
     ├─ Parse body: JSON string
     └─ Return: HttpRequest struct

  2. ANALYZE (scred-http-detector::HttpContentAnalyzer)
     ├─ Analyze headers:
     │  ├─ Authorization: "Bearer token..." → Sensitivity::Secret
     │  ├─ Content-Type: "application/json" → ContentType::Json
     ├─ Analyze body:
     │  ├─ Detect JSON structure
     │  ├─ Find "password" field → Sensitivity::Secret
     │  ├─ Find "username" field → Sensitivity::Internal
     └─ Return: AnalysisResult with findings

  3. REDACT (scred-http-redactor::Http11Redactor)
     ├─ Redact headers:
     │  └─ Authorization value → "[REDACTED]"
     ├─ Redact body (JSON):
     │  ├─ "password" value → "[REDACTED]"
     │  ├─ Keep "username" (per redaction rules)
     │  └─ Result: {"username": "john", "password": "[REDACTED]"}
     └─ Return: Redacted HttpRequest

  4. FORWARD to upstream
     └─ Upstream sees redacted data only

  5. Response comes back (e.g., {"token": "abc123..."})

  6. ANALYZE response
     └─ Detect "token" field is sensitive

  7. REDACT response
     └─ "token" value → "[REDACTED]"

  8. SEND to client
     └─ Client gets: {"token": "[REDACTED]"}

═══════════════════════════════════════════════════════════════════════════════
 KEY DESIGN DECISIONS
═══════════════════════════════════════════════════════════════════════════════

1. SEPARATION OF CONCERNS ✅
   Detection (analyze what) ≠ Redaction (do how) ≠ Protocol (handle structure)
   Each crate has ONE clear responsibility

2. NO H2 IN REDACTOR ✅
   ✓ h2_adapter renamed to H2Redactor, moved to scred-http-redactor
   ✓ scred-redactor stays protocol-agnostic
   ✓ H2 concerns isolated in HTTP-specific layer
   ✓ Respects your constraint exactly

3. PROTOCOL FLEXIBILITY ✅
   ✓ Easy to add HTTP/3 (new Http3Redactor)
   ✓ Easy to add WebSocket (new WsRedactor)
   ✓ Each protocol gets own strategy module
   ✓ Shared infrastructure (HeaderRedactor, BodyRedactor)

4. COMPOSABILITY ✅
   ✓ Http11Redactor = HeaderRedactor + BodyRedactor
   ✓ H2Redactor = HeaderRedactor + BodyRedactor (same components)
   ✓ Analyzers follow same pattern
   ✓ Easy to test components independently

5. REUSABILITY ✅
   ✓ scred-http-detector: Can be used by any HTTP analyzer
   ✓ scred-http-redactor: Can be used by any HTTP proxy
   ✓ No MITM/proxy specific code in reusable layers
   ✓ Each crate is independently valuable

6. TESTABILITY ✅
   ✓ Mock HttpContentAnalyzer for testing redactor
   ✓ Mock HttpRedactor for testing detector
   ✓ Unit test each layer independently
   ✓ Easy to create test doubles
